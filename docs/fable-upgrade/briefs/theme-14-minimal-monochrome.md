## 1. DNA 요약

**"무스타일의 스타일" — 색이 사라질 때 남는 것. 침묵·집중·중립의 순수 무채색 시스템.**

명시된 규칙 (README.md:13-23, tokens.css 주석, semantic.css 주석 기준):

| 규칙 | 근거 |
|------|------|
| **유채색 0** — 순흑~순백 16단계 그레이 램프(R=G=B)만 존재. 컬러 램프 자체가 없음 | tokens.css:14-34, README.md:15 |
| 위계는 색이 아니라 **명도·굵기(300/400/500/600)·크기·여백**으로만 | tokens.css:119-122, README.md:16 |

| **광활한 여백** — space 스케일 끝단 256px, `.section` 96px 수직 리듬 | tokens.css:51-75, base.css:225-227 |

| **FORBIDDEN에 준하는 명시**: "Noise/grain 텍스처는 쓰지 않는다 — 헤어라인과 여백만으로 깊이를 만든다" | base.css:323-324 주석 |

## 2. 강점 Top 3 (살리고 더 밀어붙일 것)

### ① 토큰 아키텍처의 규율이 실제로 지켜짐 — 무채색 순도 100%
- 전 CSS를 grep한 결과 컴포넌트 레이어에 하드코딩 hex 0건, 유채색 0건. `tokens → semantic → components` 3층 경계가 실제로 작동한다 (components/*.css 전체, CHECKLIST.md:13-14 검증 기록과 일치).
- 상태색까지 무채색 명도로 재해석한 `--color-success/warning/danger/info` 4벌 세트(semantic.css:50-59)는 30개 테마 중에서도 드문 개념적 완성도. **→ 이 순도를 마케팅 포인트로 인덱스에서 더 극적으로 전시할 것.**

### ② "명도 사다리" 자체가 브랜드 비주얼이 될 잠재력
- 16단계 램프를 JS로 자동 생성해 전시하는 `#rampGrid`(index.html:114, 556-571), 그레이스케일 전용 차트 시스템(`--chart-ink-1~4` + 점선/해치 패턴으로 시리즈 구분, charts.css:39-72), 무채색 기여 그리드(profile.html)까지 — 명도만으로 데이터를 말하는 언어가 이미 구축돼 있다. **→ 램프를 '전시물'에서 '공간 연출'로 승격시킬 것 (5장 참조).**

### ③ 인터랙션 인프라가 프로덕션급
- 포커스 트랩(app.js:42-52), ⌘K 커맨드 팔레트, ESC/화살표 키보드 내비, `aria-sort` 실동작(app.js:495-498), reduced-motion 전면 무력화(base.css:346-354), 스킵링크 전 페이지 적용. 칸반은 카드에 tabindex+role+Enter/Space까지 부여(kanban.html:771-782).
- 다크모드가 단순 반전이 아니라 "그림자→보더 대체"(semantic.css:125-128) 같은 재해석을 함. **→ 이 수준을 유지한 채 시그니처 모션만 얹으면 된다.**

## 3. AI 지문 (테마 DNA와 무관하게 배어든 것)

크림 톤·세리프 남용 같은 '서재 관성'은 이 테마에 **거의 없다** (무채색 규율이 막아줌). 남은 지문은 구조·관성 쪽이다.

### 3-1. 죽은 세리프 임포트 — 서재 관성의 화석
- **tokens.css:10** — `@import`가 `Noto Serif KR`(400/500/600/700)을 로드하지만 **전 코드베이스에서 세리프 사용 0건** (grep 확인). 30개 테마 일괄 폰트 전략의 잔재가 그대로 남은 전형적 AI 흔적 + 무의미한 웹폰트 페이로드.
- 같은 줄의 `Noto Sans KR`도 `--font-mono` 스택(tokens.css:86)에만 등장하는 반쪽 사용. **→ Noto Serif KR 임포트 삭제, Noto Sans KR 정리.**

### 3-2. hero → 스탯 카운터 → 균질 카드그리드 공식
- index.html:83-101 — overline → 대형 헤드라인 → lead → 버튼 3개 → 숫자 스탯 4개(16/60+/9/0) 나열은 AI 랜딩의 표준 공식. ("0 유채색" 카피 자체는 좋음 — 구조가 문제.)
- index.html:451-461 — 9개 페이지 링크가 3×3 균일 카드 그리드로 진열. `.page-link` 전부 동일 크기·동일 밀도.
- pricing.html:46-50 — 중앙정렬 히어로 + 3열 플랜 그리드 + "가장 인기" 배지(94-95행)도 교과서 공식 그대로.

### 3-3. 컴포넌트 진열대 페이지
- index.html:167-442 — 60+ 컴포넌트가 동일 폭 1열 `demo-card` 스택으로 무맥락 나열. 디자인시스템 허브라 갤러리는 필요하지만, 현재는 '진열대'지 '타입 스페시멘 도록'이 아니다. 밀도·스케일 변화 0, 모든 demo-card가 같은 라운드·같은 패딩·같은 라벨 포맷.

### 3-4. 균질한 등장 모션
- base.css:329-341 — `.reveal` fade-up(8px) + 50ms 간격 스태거는 모든 AI 산출물이 쓰는 기본기. 테마 고유 모션 언어가 아님.

### 3-5. 빈 껍데기 푸터 + dev 문서 링크 노출
- product.html:245-250, 404.html:90-95 등 — 브랜드마크 + © 한 줄뿐인 무성의 푸터가 페이지마다 복붙.
- index.html:69(`href="README.md"`), index.html:473-474(`README.md`/`CHECKLIST.md` 링크) — 브라우저에서 raw 마크다운이 열리는 dev-tool 흔적. 프로덕션 사이트라면 있을 수 없는 링크.

## 4. 프로덕션 결함 (파일:라인)

### A. 기능/정합성 버그
1. **product.html:79-85 — 별점 표시 모순.** `aria-label="평점 5점 만점에 4.5점"`, `data-rating="4"`인데 별 5개가 전부 `is-filled`. 시각은 5.0, 라벨은 4.5, 데이터는 4. 하프스타 또는 4개 채움으로 정합화 필요.
2. **product.html:224 — 추천 카드의 별점이 별 1개만 렌더** (`aria-label="4점"`인데 filled 별 1개). 숫자 병기("4.0")를 하든 별 4개를 넣든 통일 필요.
3. **product.html:221-240 — `.card-interactive`(cursor:pointer, hover 보더 강조)인 추천 상품 카드가 `<a>`도 아니고 클릭 핸들러도 없음.** 클릭 유도 어포던스만 있고 동작·키보드 접근 모두 불가. 링크로 감싸야 함.

### B. ARIA 오용 (색 의존 0을 자랑하는 테마라 더 아픈 부분)
4. **index.html:251-254** — `role="tablist"` 컨테이너 안의 `.seg` 버튼들에 `role="tab"` 없이 `aria-selected`만 부여. 일반 button의 `aria-selected`는 무효 ARIA. (dashboard.html:199-204는 올바르게 `role="tab"` 부여 — 페이지 간 불일치.)
5. **settings.html:105** — segmented에 role 자체가 없는데 `aria-selected` 사용. 같은 무효 패턴.
6. **product.html:103-107** — `role="radiogroup"`의 자식 버튼이 `role="radio"`+`aria-checked`가 아니라 `aria-selected` 사용. 스크린리더에 선택 상태 전달 안 됨.
7. **index.html:260-264** — 레이팅 별 중 첫 번째만 `role="radio" aria-checked` 보유, 나머지 4개는 role 없음. 전부 통일 필요.

### C. 토큰 우회/미정의 참조
8. **semantic.css:71** — `var(--neutral-920, var(--neutral-900))`: `--neutral-920`은 tokens.css에 존재하지 않음. fallback으로 동작은 하지만 죽은 참조. `--neutral-900`으로 정리하거나 920 단계를 정식 추가.
9. **inbox.html:107** — `var(--color-scrim, rgba(0,0,0,.4))`: `--color-scrim` 미정의. semantic.css의 `--color-overlay`가 정확히 이 용도(semantic.css:48)이므로 교체.

### D. 한글 웨이트 공백 (타이포 품질)
10. **tokens.css:10** — Gothic A1을 400/500/700만 로드. 그러나 시스템은 `--weight-light(300)`과 `--weight-semibold(600)`을 핵심 위계로 사용:
    - index.html:33 `.hero-display .thin`(300) — 히어로 "색이 아닌 빛으로."가 **한글이라 300이 없어 400으로 폴백** → 라이트/세미볼드 대비라는 히어로의 핵심 연출이 한글에서 무너짐 (Inter는 300 로드되어 있어 영문만 얇게 나옴).
    - 600 요청 시 Gothic A1은 700으로 매핑되어 한글 semibold가 영문(Inter 600)보다 무겁게 렌더. **→ Gothic A1 300;600 추가 로드 (혹은 wght 축 지정).**

### E. 상태/기타
11. **components/forms.css** — 입력은 `:focus` 스타일만 있고 마우스/키보드 구분 없음(27-31행). 입력 필드는 관례상 허용 범위지만, `.fileupload` 드롭존·`.pd-thumb`(product.html:22-25) 등 커스텀 컨트롤에 `:focus-visible` 스타일 부재 — base.css 전역 링(95-99)에만 의존. 커스텀 포커스가 있는 checkbox/switch(selection.css)와 일관성 맞출 것.
12. **kanban.html** — HTML5 DnD에 키보드 대체 수단 없음(카드 '이동'은 마우스 전용). CHECKLIST.md:77에서 스스로 인지하고 있는 한계. 카드 모달에 "컬럼 이동" 액션을 추가하면 해소 가능.
13. **SVG var() 버그: 해당 없음** — 전 HTML에서 `stroke="var(..)"`/`fill="var(..)"` 속성 사용 0건 (전부 class 기반 CSS 스타일링). 통과.
14. **깨진 링크류**: index.html:69의 `README.md` 내비 링크(3-5절 참조). 그 외 오버레이 타깃 ID는 실재 확인됨.

## 5. 대담화 기회 (프로 디자이너라면 밀어붙일 지점)

> 원칙: 색·노이즈·질감 추가는 금지(DNA). 무기는 **명도, 스케일, 반전, 헤어라인, 모션** 5가지뿐이며, 그래서 더 과감해야 한다.

### ① "블랙 반전 섹션" — 갤러리 화이트 속의 순흑 전시실
현재 인덱스는 처음부터 끝까지 흰 배경 한 장(index.html 전체). 미술관도 전시실마다 벽 색을 바꾼다. 페이지 중간에 **풀블리드 순흑(`--neutral-0`) 섹션**을 삽입해 화이트-온-블랙으로 반전시킬 것 — 예: Foundations의 명도 램프 섹션, 또는 "철학" 섹션을 통째로 흑막 처리. 라이트 페이지 안에서 다크 토큰 셋을 국소 적용(`data-theme="dark"`를 섹션에 거는 것만으로 가능 — 이미 변수 스왑 구조가 있음)하면 구현 비용도 낮다. 이 한 방이 "0→1000 명도 스펙트럼을 오간다"는 테마 서사를 몸으로 보여준다.

### ② 명도 램프의 공간화 — 스와치에서 환경으로
`#rampGrid`(index.html:114)는 얌전한 스와치 표. 이걸:
- 히어로 배경에 **16단계 세로 헤어라인 스트라이프** 혹은 화면 하단에서 순흑→순백으로 이어지는 **풀폭 램프 밴드**로 확장,
- 404 페이지의 거대 `404`(404.html:17, clamp 최대 14rem — 이 스케일 감각은 이미 좋음)를 **아웃라인 스트로크 텍스트**(`-webkit-text-stroke: 1px` 헤어라인 타이포)로 바꿔 "빈 페이지=빈 글자" 컨셉 완성,
- 스크롤에 따라 섹션 배경 명도가 975→950→앵커 지점 순흑으로 계단식 하강하는 연출.

### ③ 시그니처 모션: "잉크가 차오르는" 인터랙션
현재 호버는 전부 명도 스왑뿐(buttons.css:54, 63). 무채색 테마의 시그니처는 **블랙 잉크 필**이어야 한다:
- primary/danger 버튼 호버 시 배경이 좌→우 혹은 하→상으로 차오르는 트랜지션(`background-size`/`transform: scaleY` 기법, 200ms `--ease-out`),
- 내비/링크 호버는 헤어라인이 자라나는 언더라인(현재 `text-decoration-color` 스왑 → `background-image` 언더라인 width 애니로 교체),
- 테마 토글 시 View Transition API로 **원형 마스크가 화면을 삼키는 흑↔백 전환** (지원 브라우저 한정, progressive). reduced-motion 대응은 기존 인프라 재사용.

### ④ 컴포넌트 갤러리를 '타입 스페시멘 도록'으로
균일한 demo-card 스택(index.html:167-442) 대신, 인쇄 도록처럼:
- 각 섹션 앞에 **거대한 섹션 폴리오**(01, 02… `--text-6xl` 이상, `--weight-light`)와 미술관 작품 라벨 형식 캡션(컴포넌트명 / 상태 수 / 사용 토큰)을 배치,
- 버튼·배지 같은 소형 컴포넌트는 조밀한 멀티컬럼, 테이블·차트는 풀폭으로 — **밀도의 완급**을 만들 것,
- 대표 컴포넌트 1-2개(버튼, 커맨드 팔레트)는 실사용 맥락 비네트(미니 장면)로 승격.

### ⑤ 헤어라인 건축 도면 배경
noise 금지 규칙(base.css:323) 안에서 허용되는 유일한 '분위기'는 헤어라인이다. 히어로/섹션 전환부에 `--color-border-subtle`의 **모눈(graph paper) 혹은 십자 크로스헤어 마커** 배경을 극히 옅게 깔면(예: 24px 그리드, opacity 그대로 토큰 명도로) '설계도 위의 시스템'이라는 서사가 생긴다. 도트나 텍스처가 아닌 1px 선이므로 DNA 합치.

### ⑥ 히어로 타이포 극단화
`clamp(3rem, 11vw, 7rem)`(index.html:32)은 아직 얌전하다. 12-14vw까지 키우고, 300 웨이트 한글 로드(4-D-10 수정과 연동)로 **초박(超薄) vs 세미볼드의 병치**를 진짜로 작동시켜라. 좌측 정렬 유지하되 두 번째 줄을 우측으로 크게 들여쓰는 비대칭 조판이면 "조용한 위계"가 구도로도 표현된다.

## 6. 한글 폰트 페어링

| 항목 | 현재 상태 | 판정 |
|------|-----------|------|
| 본문/UI 산세 | `Inter`(영문, 300/400/500/600 link 로드) + `Gothic A1`(한글, 400/500/700 @import) + Apple SD Gothic Neo/Pretendard 폴백 — tokens.css:82-84 | **보존.** 기하학적 중립 산세 페어링으로 DNA(휴머니스트/뉴트럴 산세)와 정합. 단 **Gothic A1에 300·600 웨이트 추가 로드 필수** (4-D-10) |
| 모노 | SF Mono/JetBrains Mono/… + `Noto Sans KR` 폴백 — tokens.css:85-86 | 유지 가능. 단 mono 스택 내 Noto Sans KR은 한글 글리프 폴백 용도임을 주석으로 명시 권장 |
| **Noto Serif KR** | tokens.css:10에서 4개 웨이트 로드되나 **사용처 0** | **삭제 대상.** DNA 근거 없는 서재풍 세리프의 잔재(임포트만 남은 화석). 페이로드 낭비 |
| Noto Sans KR | @import 로드되지만 `--font-sans`에는 미포함(mono 폴백에만) | 정리 대상 — mono 폴백으로만 쓸 거면 유지하되, 아니면 임포트에서 제거 |

**결론**: 페어링 자체(Inter × Gothic A1)는 이 테마의 정답에 가깝다. 교체가 아니라 **웨이트 정합(300/400/500/600)과 죽은 세리프 제거**가 과제.
